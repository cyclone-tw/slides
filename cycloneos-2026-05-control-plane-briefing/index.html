七日改造報告

CycloneOS 正在從「本機工具」升級成「雲端指揮中心」

簡單說：以後不是每個工具各自做各自的事，而是由 Dashboard 統一看進度、留紀錄、做審核，再把真正的工作交給 OpenAB、雲端服務，或 Mac mini 上的 OpenClaw 去執行。

會議重點：這不是單純換 UI，也不是把 Discord bot 搬家，而是在建立一套「可追蹤、可控管、可替換」的 AI 工作系統。

先用白話講清楚

過去比較像是「很多工具散在不同地方」：有 Dashboard、有 Discord bot、有本機腳本、有 Mac mini 的 OpenClaw。這週的工作是在把它們整理成一套分工明確的系統。

你提出需求在 Dashboard 或未來的 Discord / API 輸入任務。
Dashboard 記錄誰提出、要做什麼、目前狀態、是否需要審核。
OpenAB 分派判斷這件事要交給雲端 AI、雲端 API，還是 Mac mini。
Worker 執行真正做事的是 Codex / Claude / OpenClaw / Drive / Notion 等工作者。
回寫結果結果、錯誤、檔案與紀錄回到 Dashboard。

OpenAB 現在做到哪裡

已完成，可以當作進度亮點

  • 已把 OpenAB 原始碼放進 CycloneOS 的服務區。
  • 已做出 CycloneOS 專屬的 Zeabur 部署方式。
  • 服務可以啟動，健康檢查成功。
  • 已能接收一個測試任務，並把它放進待處理狀態。
  • 這個新 OpenAB 與舊有 OpenAB 分開，不是直接改壞舊服務。

還不能說正式完成

  • 還沒有公開網址，所以 Dashboard 尚未正式切過去。
  • 任務目前只放在記憶體，服務重啟會不見。
  • Mac mini worker 還不能主動領任務、回報進度、回傳結果。
  • 允許做哪些動作還需要更嚴格的規則。
  • 測試用服務需要最後清理。

還缺什麼

這些不是失敗，而是下一階段要補上的缺口。會議時可以直接用這張當「風險與待辦」頁。

OpenAB 正式網址

沒有穩定公開網址，就不能讓 Dashboard 正式切到新 OpenAB。

任務要寫進資料庫

現在新 OpenAB 的任務還在記憶體，必須改成重啟後也不會消失。

Mac mini 還沒接上

Draco / OpenClaw 還不能從 OpenAB 領任務並回報結果。

產出檔案要有家

報告、截圖、匯出檔需要穩定放在雲端儲存位置。

流程介面要整理

Documents / YouTube / Social 比較適合變成流程卡，不只是聊天頻道。

工作者治理

Codex 監督 Claude worker 的模式已經形成，接下來要產品化成規則。

下一步建議

0-7 天：先接通主幹

  • 幫新 OpenAB 綁定公開網址。
  • 讓 Dashboard 正式改用 CycloneOS 專屬 OpenAB。
  • 讓 OpenAB 能保存任務、派任務、收回報。
  • Mac mini 先支援 ping、status、最近 log。

2-4 週：讓它可用

  • 任務紀錄完整進 Postgres。
  • Work History 可以看任務細節與結果。
  • 加入需要確認才執行的 approval。
  • 把產出檔案放到穩定雲端儲存。

1-2 個月：變成工作系統

  • 用 React Flow / SkillGraph 管理流程。
  • Discord 變成入口，不再是獨立執行核心。
  • 長期記憶、摘要、決策紀錄進系統。
  • Codex、Claude、OpenClaw 都能當可替換 worker。
會議結論建議：先把「Dashboard → 新 OpenAB → Mac mini」這條線接通。接通後，後面所有自動化都能進同一套紀錄、審核與安全規則。
